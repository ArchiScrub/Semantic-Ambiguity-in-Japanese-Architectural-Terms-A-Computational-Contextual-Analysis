# Semantic-Ambiguity-in-Japanese-Architectural-Terms-A-Computational-Contextual-Analysis
Public repository for a BERT-based contextual analysis of semantic ambiguity in Japanese architectural terms. It contains code, configs, outputs, logs, manifests, and figures for testing eight terms across FULL and CLEAN corpora through screening audit, polysemy recovery, register alignment, sense-aware interpretation, and an engawa boundary probe.

Paper materials

This repository contains the supplementary materials, analysis script, configuration files, runtime log, manuscript figures, supplementary figures, and generated results for the paper:

**Semantic Ambiguity in Japanese Architectural Terms: A Computational Contextual Analysis**

The study uses Natural Language Processing (NLP), specifically a fine-tuned multilingual BERT masked language model, to examine semantic ambiguity in selected Japanese architectural terms. The analysis tests whether ambiguity remains recoverable across paired **FULL** and **CLEAN** corpus conditions.

The study focuses on eight Japanese architectural terms:

```text
engawa
tokonoma
ma
mu
shakkei
doma
shōji
fusuma
```

The main workflow combines screening audit, polysemy recovery, architecture-facing register alignment, sense-aware interpretation, and a bounded `engawa` concept-alignment test.

## Repository Description

This repository preserves the implementation record and generated materials for a BERT-based contextual analysis of semantic ambiguity in Japanese architectural terms. It includes the main script, YAML configuration files, filters, runtime log, edited manuscript figures, supplementary figures, CSV outputs, provenance files, master results, and the bounded `engawa` concept-alignment outputs across paired FULL and CLEAN corpus conditions.

## Data Source

The pre-processed corpus framework used for this study is based on the preceding repository associated with the earlier paper:

```text
Reproducibility-and-Validation-of-a-Computational-Framework-for-Architectural-Semantics
```

That preceding repository contains the source corpus materials and related framework files used to prepare the FULL and CLEAN corpus conditions.

For the present repository, the script expects the BERT corpus files in the following structure:

```text
txt_corpus/
  BERT_Corpus.txt
  BERT_Corpus_CL.txt
```

The **FULL** corpus preserves the broader architecture-facing expository field. The **CLEAN** corpus removes glossary-like and explicitly definitional sentences in order to test whether semantic ambiguity remains detectable after direct definitional support is reduced.

## Analytical Workflow

The analysis follows five main stages.

### 1. Screening Audit

The script checks whether each target term has enough usable sentence-level target contexts under both FULL and CLEAN corpus conditions.

The screening audit uses thresholds for:

```text
minimum FULL contexts
minimum CLEAN contexts
target-span recovery
polysemy stability
CLEAN non-collapse
fallback ratio
FULL/CLEAN k-shift
```

### 2. Polysemy Recovery

The script extracts contextual BERT embeddings for target-term occurrences and applies conservative binary clustering to test whether recurring internal semantic structure is recoverable.

The reported Paper 4 configuration caps automatic polysemy recovery at:

```text
k = 2
```

This means the recovered clusters should be read as a conservative internal contrast, not as a complete dictionary-style sense inventory.

### 3. Architecture-Facing Register Alignment

The recovered term profiles are aligned with architecture-facing interpretive registers.

The main registers are:

```text
physical-spatial
symbolic-theoretical
represented/imagined
boundary/transition
hybrid
```

The classification layer also includes secondary relation cues, such as boundary or edge conditions, threshold or transition, liminality, permeability or weak enclosure, display or ritual framing, and visual incorporation or borrowed scenery.

### 4. Sense-Aware Interpretation

The script reloads the recovered cluster centroids and cluster sizes to interpret term-internal variation.

This stage avoids reducing each target term to a single averaged semantic profile. Instead, it compares recovered sense centroids against the same architecture-facing prompt vectors used in the register-alignment stage.

### 5. Bounded `engawa` Concept-Alignment Test

A focused secondary module is applied only to `engawa`.

This module tests whether `engawa` aligns with:

```text
boundary-edge logic
connector-flow logic
derived hybridity
river-bank analogy
```

This test is used as a bounded architectural hypothesis-stress test. It does not claim to prove etymology or historical origin.

The FULL/CLEAN contrast is used to test whether ambiguity remains recoverable after overt definitional scaffolding is reduced.

## How to Inspect the Results

To inspect the reported results without rerunning the pipeline:

1. Download the repository.
2. Extract the `.rar` archives.
3. Open `results_paper4.rar`.
4. Inspect `summary_dashboard.csv` and `screening_decision_table.csv`.
5. Check the term-specific `polysemy_representatives` and `polysemy_scores` CSV files.

## Reproducibility Notes

The run uses deterministic controls where possible, including:

```text
seed: 42
kmeans_n_init: 50
kmeans_max_iter: 1000
```

The script also sets deterministic PyTorch behavior, disables cuDNN benchmarking, and disables TF32 fast paths.

Exact numerical replication may still depend on local hardware, CUDA version, PyTorch version, Transformers version, and the exact model checkpoint files.

The runtime log records one audit warning:

```text
CLEAN is not an exact sentence-level subset of FULL after normalization/splitting: 201 CLEAN sentences not found in FULL.
```

This warning is preserved in the repository for transparency.
